## Context

當前專案需要開發一個 Web 介面來與藍牙硬體產品連線並進行測試。現有的藍牙操作工具多為終端機指令（CLI）形式，這對非軟體工程背景的測試人員或客戶不夠友善。本設計旨在建立一個簡單直觀的網頁控制面板，整合設備掃描、過濾、Service UUID、Write/Read/Notify Character UUID 設定，以利快速進行讀寫測試。並且需要解決以往工具（如以往使用 Visual Studio UWP）常掃描到快取或未上電設備的痛點。

## Goals / Non-Goals

**Goals:**
- 提供網頁化控制面板，列出周遭藍牙設備（不需預先檢查 Service UUID 或配對）。
- 提供依據 Name 或 MAC Address 的過濾設定，並將設定持久化儲存（例如使用 LocalStorage）。
- 確保列出的設備是「真實上電且活躍」的，排除系統快取的無效設備。
- 支援從列表選擇設備並與特定的 Service UUID 建立連線。
- 支援啟用 Read UUID 的 Notify 功能。
- 提供指令寫入區塊（寫入 Write UUID），支援 Binary（如 `FF FF AF`）與 Text 模式切換。
- 實作字元輸入事件攔截，每個輸入字元均可觸發特定 Function，保留高度客製化邏輯的彈性。
- 顯示互動日誌與狀態資訊。

**Non-Goals:**
- 支援多個藍牙裝置同時連線（本階段僅針對單一裝置進行連線測試）。
- 提供複雜的使用者帳號管理與雲端設定同步（目前僅在地端儲存設定）。

## Decisions

### 決策 1：後端藍牙引擎與 API 架構
- **選擇方案**：使用 Python FastAPI 作為網頁後端，底層採用 bleak 進行藍牙互動。
- **評估替代方案**：使用瀏覽器內建之 Web Bluetooth API。
- **決定原因**：雖然 Web Bluetooth API 可以直接在前端執行，但其無法取得完整的 MAC Address 進行過濾，且掃描機制的快取行為較難由開發者直接控制。採用 Python FastAPI + bleak 的架構，不僅跨平台支援度佳，更能透過後端邏輯更好地處理藍牙廣播封包的即時性，藉由檢查 RSSI 更新或廣播時間戳記來過濾掉未上電的快取設備。

### 決策 2：前後端即時資料流傳輸方式
- **選擇方案**：使用 WebSocket 傳輸即時掃描結果與 Notify 通知，配合 REST API 進行寫入（Write）、讀取（Read）與連線狀態控制。
- **決定原因**：藍牙掃描與通知（Notify）都是由設備端主動發送的非同步事件。WebSocket 允許後端在收到廣播封包或 notification callback 時，即時推送至網頁前端，確保低延遲的呈現。

### 決策 3：前端過濾儲存與客製化指令輸入
- **選擇方案**：前端使用 `localStorage` 儲存設備的 MAC Address 或 Name 過濾條件。指令輸入區實作事件監聽器，每輸入一個字元即呼叫一個 hook function，並提供 UI 切換 Binary（如十六進位表示）與 Text 模式。
- **決定原因**：`localStorage` 輕量且無需後端資料庫介入。字元層級的事件觸發設計可以讓使用者在未來輕易掛載自定義的程式邏輯（例如動態組合特定通訊協定的封包）。

### 決策 4：排除快取與未上電設備
- **選擇方案**：在後端進行掃描時，不完全依賴作業系統的靜態快取清單。要求掃描機制必須在指定時間窗口內實際收到該設備的 Advertisement 封包（檢查近期是否有收到訊號），才將其列為真實活躍設備回傳給前端。

## Risks / Trade-offs

- **[風險] 徹底排除系統快取設備的難度** → **[緩解措施]**：某些作業系統（如 Windows）底層藍牙 API 具有強固的快取機制。若 bleak 預設行為無法完全避開，可能需要在掃描邏輯中實作進階過濾，例如過濾掉一段時間內未更新 RSSI 值的設備。
- **[風險] 每個字元觸發 Function 可能造成的效能問題** → **[緩解措施]**：前端架構需確保該觸發 Function 設計良好。若觸發 Function 包含發送實體藍牙封包，應給予使用者適當的警告，或預設僅作為邏輯轉換（如即時計算 Checksum），並透過獨立的送出按鈕來發送最終指令。

## UI/UX 設計決策 (UI/UX Design Decisions)

為因應產線測試與高效率除錯需求，前端介面經過以下特化設計：
- **雙分頁架構 (Tabs)**：將「主要控制 (掃描、測試面板)」與「設定與日誌 (UUID設定、互動紀錄)」分離，保持主畫面的高對比度與清爽度。
- **巨大化狀態面板 (OK/NG Display)**：在主要分頁中央配置顯眼的狀態顯示區塊，可透過 JS 函式 (`setTestResult`) 動態改變為藍色(OK)或紅色(NG)，便於遠距離觀看結果。
- **條碼機友善 (Barcode-Friendly)**：發送指令的輸入區塊（Input）放大約 40%，以容納並清楚顯示大量條碼資料。
- **快速連線機制**：掃描結果列表支援雙擊 (Double-click) 自動連線，縮短操作時間。
- **全狀態記憶**：所有的勾選、輸入與選項狀態均使用 `localStorage` 持久化，重整網頁不遺失進度。
- **隱藏無效設備**：增加「隱藏無名稱 (Unknown)」設備的選項，減少畫面雜訊。

## Development Process & Version Control

- **嚴格版控要求**：本專案的任何功能更新、架構調整與程式碼修改，都必須在每次更新段落完成時進行 Git 版本控制（Commit），以確保程式碼的歷史紀錄完整，隨時可追溯與還原。
